# JC-Packages v5.0.0

**Release date:** 17 July 2026
**Previous version:** 4.1.0

## Overview

v5.0.0 moves the multi-tenancy contract out of JC.Identity and into JC.Core. `IMultiTenancy`, `Tenant` and `TenantSettings` now live in `JC.Core.Models.MultiTenancy`, so **any package with domain models can be tenant-scoped without taking a dependency on JC.Identity**.

The release also introduces **JC.FileStorage** and **JC.FileStorage.Web** — the first packages built on the relocated contract, and currently the only place in the suite where it is used.

The namespace change is the entire breaking surface, so this is a **major** release. Per the [suite versioning policy](../../README.md#versioning-strategy), every package moves to `5.0.0`, including those with no functional changes.

## Highlights

- **`IMultiTenancy` and `Tenant` now live in JC.Core** — the contract is available to every package, not just JC.Identity's dependents.
- **JC.Identity still owns the mechanism** — `ApplyTenantQueryFilters` and `AllTenants` stay put. Tenant isolation remains something that arrives with JC.Identity.
- **New: JC.FileStorage** — tenant-scoped file storage with database-backed records, per-folder size and type limits, and audited deletion.
- **New: JC.FileStorage.Web** — `IFormFile` handling, MIME type inference, and an upload constraints tag helper.
- **No schema changes to existing packages.** JC.Communication, JC.Github, JC.Identity and JC.Core keep their tables exactly as they were in 4.1.0.

## Breaking changes

### Multi-tenancy types moved to JC.Core

`IMultiTenancy`, `Tenant` and `TenantSettings` have moved package and namespace:

| Type | Was | Now |
|------|-----|-----|
| `IMultiTenancy` | `JC.Identity.Models.MultiTenancy` | `JC.Core.Models.MultiTenancy` |
| `Tenant` | `JC.Identity.Models.MultiTenancy` | `JC.Core.Models.MultiTenancy` |
| `TenantSettings` | `JC.Identity.Models.MultiTenancy` | `JC.Core.Models.MultiTenancy` |

The types themselves are unchanged — same members, same behaviour, same tables. Any file with `using JC.Identity.Models.MultiTenancy;` will fail to compile until the using is updated.

`ApplyTenantQueryFilters` and `AllTenants` **stay in JC.Identity**. Only the models moved; the mechanism that acts on them did not.

**That is the entire breaking surface.** No entity gained or lost a column, no table changed, and no query behaviour is different. Everything else described below is additive.

## New features

### JC.FileStorage

Tenant-scoped file storage: files on disk, records in the database. See [Setup](../JC.FileStorage/Setup.md) and [Guide](../JC.FileStorage/Guide.md).

- **Folders** provide a single level of separation — no nesting, no versioning, no archiving. Registered at startup via `AddFolders`, per tenant.
- **Per-folder limits** — `MaxBytes` and `AllowedExtensions`, falling back to `FolderRegistry.DefaultMaxBytes` and `DefaultAllowedExtensions`. Enforced by `StorageService` itself, so no entry point can bypass them.
- **A 10GB ceiling** (`FolderModel.MaxAllowedBytes`) that no folder or default may exceed.
- **A permanent blocked list** — around sixty executable and script extensions that can never be stored, checked before any allow-list and not re-enableable by configuration.
- **Audited deletion** — the record is soft-deleted so an audit retains who removed a file and when; the file itself is permanently deleted. There is no restore path.
- **Depends only on JC.Core.** JC.Identity is needed only for tenant isolation; without it every file belongs to the no-tenant scope.

Adding JC.FileStorage requires a migration for its own `SavedFiles` table — see [JC.FileStorage — Apply migrations](../JC.FileStorage/Setup.md#3-apply-migrations). Nothing else in your schema changes.

### JC.FileStorage.Web

Optional ASP.NET Core integration for JC.FileStorage. Depends on JC.FileStorage and JC.Web.

- **`WebStorageService`** — `IFormFile` uploads validated against the folder's limits *before* the stream is read, and downloads carrying the MIME type and download name.
- **`FormFileHelper`** — `IFormFile` to name, extension and bytes; extension to MIME type via the framework's own mapping table.
- **`<upload-constraints>` tag helper** — renders a folder's accepted types and size limit from the same `FolderRegistry` values the server enforces, so the help text cannot drift from the rule.

## Migration guide

Upgrading a consuming application from 4.x to 5.0.0 is a namespace change and nothing else.

### 1. Update multi-tenancy usings

Replace the old namespace wherever it appears:

```csharp
// Before
using JC.Identity.Models.MultiTenancy;

// After
using JC.Core.Models.MultiTenancy;
```

The types and their members are identical, so nothing else in the file changes. If your `DbContext` extends `IdentityDataDbContext<TUser, TRole>`, tenant filtering continues to work exactly as before.

### 2. (Optional) Adopt JC.FileStorage

A new package; nothing to migrate away from. See [JC.FileStorage — Setup](../JC.FileStorage/Setup.md).

## Notes and limitations

- **Only `SavedFile` is tenant-scoped.** Making every domain model implement `IMultiTenancy` was considered and rejected: notifications, chat threads and their logs are scoped by the **user** who owns them — and that user already carries the tenant — while email logs and issue reports are deliberately **system-wide**. Adding a tenant column to those would duplicate what the user relationship already says, or wrongly partition data meant to be shared. The contract moved to JC.Core so that packages *can* opt in where it genuinely fits, not so that everything must.
- **Isolation still requires JC.Identity.** `IMultiTenancy` lives in JC.Core, but the query filter that acts on it is applied by `IdentityDataDbContext`. An application whose context does not extend it gets no filtering, and implementing the interface alone enforces nothing.
- **A null tenant is a scope, not a wildcard.** It is isolated exactly like a named tenant. Rows with `TenantId = NULL` are visible only to users with no tenant, or through a deliberate cross-tenant query.
- **`Tenant` reaches your model once you add JC.FileStorage.** `SavedFile` references it, so the entity is pulled into the model even without JC.Identity. The table is named `Tenants` under `IdentityDataDbContext` (from its `DbSet`), but `Tenant` on a JC.Core-only context (from the type name).
- **Cross-tenant access is unauthorised by the packages.** JC.FileStorage's `*ForTenant` methods bypass the query filter and perform no role check — they cannot, as `SystemAdmin` lives in JC.Identity. The consuming application must authorise every such call.
- **`AllowedExtensions` is a usability guard, not a security control.** Only the extension is compared, so a renamed file passes. The blocked list stops obvious mistakes, not a determined uploader.
