# JC-Packages v4.0.0

**Release date:** 4 July 2026
**Previous version:** 3.7.0

## Overview

v4.0.0 adds **multi-DbContext support** to the JC.Core repository layer and makes every built-in background job aware of the context it operates against. A single application — for example a central admin portal that manages several other applications' databases — can now read and write multiple databases through one `IRepositoryManager`, each with its own transactions and its own cleanup jobs.

Delivering this required a breaking change to how repositories are obtained, so this is a **major** release. Per the [suite versioning policy](../../README.md#versioning-strategy), every package moves to `4.0.0`, including those with no functional changes.

## Highlights

- **Multiple DbContexts** — `IRepositoryManager.For<TContext>()` returns a repository manager bound to any registered context.
- **No per-entity registration** — `GetRepository<T>()` creates repositories on demand for any entity type; the `RegisterRepositoryContexts` step is gone.
- **Context-aware background jobs** — every cleanup job gains a generic `<TContext>` form so it can run against any managed database, while the non-generic form (and existing registrations) continue to target the default context.
- **New** `ConfigureNotificationBackgroundJobs` in JC.Communication, matching the existing Email and Messaging configuration methods.

## Breaking changes

### Repository access (JC.Core)

Repositories are now obtained exclusively through `IRepositoryManager`:

- `RegisterRepositoryContexts(...)` and `RegisterRepositoryContext<T>()` are **removed** — calling them is now a compile error.
- `AddCore<TContext>()` no longer registers any `IRepositoryContext<T>` in DI (previously it registered `AuditModel` and `AuditEntry`).
- **Injecting `IRepositoryContext<T>` directly no longer resolves.** Inject `IRepositoryManager` and call `GetRepository<T>()` instead.

That is the entire breaking surface. The background-job and versioning changes described below are additive — existing job registrations continue to compile and run unchanged.

## New features

- **`IRepositoryManager.For<TContext>()`** — a repository manager bound to a specific `DbContext`, for reading and writing additional (managed) contexts. See [JC.Core — Multiple DbContexts](../JC.Core/Guide.md#multiple-dbcontexts).
- **On-demand `GetRepository<T>()`** — works for any class with no registration step; repositories are created on first use and cached for the lifetime of the scope.
- **Per-context transactions** — each context owns its own transaction via the manager returned by `For<TContext>()`.
- **`ConfigureNotificationBackgroundJobs`** (JC.Communication) — overrides `NotificationBackgroundJobOptions`, matching `ConfigureEmailBackgroundJobs` and `ConfigureMessagingBackgroundJobs`.
- **Context-aware background jobs** — `AuditCleanupJob`, `SoftDeleteCleanupJob`, `EmailLogCleanupJob`, `NotificationLogCleanupJob`, `ActivityLogCleanupJob`, and `ReadLogCleanupJob` each gain a generic `<TContext>` form for cleaning a specific managed database. The non-generic form is preserved and targets your default context, so **existing job registrations need no change**.
- **Generic-aware Hangfire job IDs** — the default recurring-job ID for a generic job now includes its type arguments (e.g. `AuditCleanupJob<AppDbContext>` registers as `"AuditCleanupJob(AppDbContext)"`), so the same job stays unique when registered against multiple contexts. Non-generic registrations keep their original ID.

## Migration guide

Follow these steps to upgrade a consuming application from 3.x to 4.0.0.

### 1. Remove `RegisterRepositoryContexts` calls

`AddCore` is now the only JC.Core registration you need.

```csharp
// Before
builder.Services.AddCore<AppDbContext>();
builder.Services.RegisterRepositoryContexts(typeof(Product), typeof(Order));

// After
builder.Services.AddCore<AppDbContext>();
```

### 2. Replace direct `IRepositoryContext<T>` injection with `IRepositoryManager`

```csharp
// Before
public class ProductService(IRepositoryContext<Product> products)
{
    public Task<Product?> GetAsync(int id) => products.GetByIdAsync(id);
}

// After
public class ProductService(IRepositoryManager repositories)
{
    private readonly IRepositoryContext<Product> _products = repositories.GetRepository<Product>();

    public Task<Product?> GetAsync(int id) => _products.GetByIdAsync(id);
}
```

Existing background-job registrations need no change — the non-generic cleanup jobs still target your default context.

### 3. (Optional) Adopt managed contexts

If your application hosts more than one database, register the extra contexts, address them with `For<TContext>()`, and register cleanup jobs against them with the closed generic form:

```csharp
builder.Services.AddMySqlDatabase<PortfolioDbContext>(builder.Configuration,
    migrationsAssembly: "Portfolio.Data",
    connectionStringName: "PortfolioConnection");

// Clean the managed context's audit trail from this application:
builder.Services.AddHangfireJob<AuditCleanupJob<PortfolioDbContext>>();

// ...and read or write it at the call site:
var projects = repositories.For<PortfolioDbContext>().GetRepository<Project>();
```

See [JC.Core — Multiple DbContexts](../JC.Core/Setup.md#multiple-dbcontexts--managed-contexts) for the full walkthrough.

## Notes and limitations

- **No distributed transactions.** A transaction is scoped to a single context; writing to two databases in one logical operation requires a transaction per context, committed independently.
- **Services stay on the default context.** Only background jobs are generic on `TContext`. Services within the packages continue to operate against the context registered with `AddCore`. Targeting a managed context from your own services is done explicitly with `For<TContext>()`.
- **Audit trails are per context.** When you write to a managed context, that context's database receives the audit entry, stamped with the current `IUserInfo` user.
